# terratheme — Architecture & Roadmap

> Multi-source-color theme generator for Terra DE.
> Replaces matugen. Phase 2 of the Terra DE project.

## What It Replaces

[matugen](https://github.com/InioX/matugen) is the current tool for generating Material 3 color palettes from wallpapers. It works, but has limitations:

1. **Single source color** — picks one dominant color from the image, ignores the rest. A wallpaper often has multiple significant hues.
2. **Limited palette variety** — only Material You variants (vibrant, expressive, tonal, etc.). No custom "moods".
3. **Template system works but is inflexible** for multi-color-source generation.

terratheme replaces matugen entirely, including all of its template-based output generation.

```
          ┌──────────────┴──────────────┐
          ▼                             ▼
┌──────────────────┐    ┌──────────────────────────┐
│  Part 1:         │    │  Part 2:                 │
│  Palette Gen     │    │  Templating Engine        │
│                  │    │                          │
│  Image →         │    │  palette.json → 12       │
│  k-means →       │    │  hardcoded targets       │

│  ♯ config needed │    │  ♯ Templates in Python  │
└──────────────────┘    └──────────────────────────┘
```

### Part 1: Palette Generation

**Input:** wallpaper image path, `--mode` (light/dark/auto), `--type` (vibrant/dim/high-contrast/muted/pastel/drawn)

**Pipeline:**

```
Image → resize → k-means (3-5 clusters) → dominant colors
  │
  ├─ Color 1 → HCT → Tonal Palettes → primary role family
  ├─ Color 2 → HCT → Tonal Palettes → secondary role family
  └─ Color 3 → HCT → Tonal Palettes → tertiary role family
       │
       ▼
  Combine → contrast validation → palette.json
```

**Key dependencies:**
- `scikit-learn` or `Pillow` for k-means clustering
- `material-color-utilities` (Python package) for HCT color space & Material 3 role math — same algorithm matugen uses under the hood, Apache 2.0 licensed
- `colormath` or manual delta-E for perceptual distance metrics

### Part 2: Templating

All templates hardcoded as Python f-strings or Jinja2 templates embedded in the code. Each target is a class:

```python
class FootTemplate(TemplateTarget):
    name = "foot"
    output_path = "~/.config/foot/colors.ini"
    description = "Foot terminal color scheme"

    def render(self, palette: PaletteData) -> str:
        return f"""# Auto-generated by terratheme
[colors-dark]
background={palette.dark.surface}
foreground={palette.dark.on_surface}
..."""
```

No `config.toml` — terratheme is opinionated. All output targets and their paths are hardcoded.

## Multi-Source Color Strategy

### How Multiple Colors Get Used

The Material 3 system already gives each role family (primary, secondary, tertiary) its own independent hue. terratheme assigns different extracted source colors to different hue slots:

| Source Color | Role Family | Slots |
|-------------|-------------|-------|
| Color 1 (most dominant) | Primary | primary, on_primary, primary_container, on_primary_container, primary_fixed, primary_fixed_dim, on_primary_fixed, on_primary_fixed_variant |
| Color 2 (second dominant) | Secondary + Surface | secondary, on_secondary, secondary_container, on_secondary_container, secondary_fixed, secondary_fixed_dim + surface, on_surface, surface_variant, on_surface_variant, background, on_background, surface_container* |
| Color 3 (third dominant) | Tertiary + Outline | tertiary, on_tertiary, tertiary_container, on_tertiary_container, tertiary_fixed, tertiary_fixed_dim + outline, outline_variant, surface_tint, scrim, shadow |

This maps to how M3 was designed — each role family has its own hue. The original Material You spec envisioned multiple source colors but Google only exposed a single-source API.

### Number of Dominant Colors

**3** is the sweet spot — maps directly to primary/secondary/tertiary role families. We could extract 4-5 but only use the top 3, keeping extras for future use.

### Light / Dark / Auto Mode

- `--mode light` — force light palette generation
- `--mode dark` — force dark palette generation
- `--mode auto` (default) — analyze image average luminance; if > 0.5 use light palette, else dark. Could also do two-pass (generate both, pick the one with better per-image contrast metrics).

## CLI

```
terratheme apply <image>              # Generate palette + render all targets
terratheme apply <image> --mode dark  # Force dark mode
terratheme apply <image> --type dim   # Dim palette type
terratheme apply <image> --dry-run    # Show what would be written
terratheme generate <image>           # Just palette.json to stdout
terratheme list-types                 # Show available palette types
terratheme list-targets               # Show output targets + paths
```

## Palette Types

| Type | Description |
|------|-------------|
| `vibrant` | Saturated, lively (like matugen's scheme-vibrant) |
| `dim` | Muted, low-contrast |
| `high-contrast` | Maximum readability |
| `muted` / `pastel` | Soft, gentle |
| `drawn` | TBD — see open questions |

## Integration with terra-packages

- `terra-de/terratheme` repo has its own CI (tests, lint)
- On tag/release (e.g., `v0.1.0`), GitHub creates a release with source tarball
- terra-packages PKGBUILD for terratheme picks up the release, builds the Arch package, deploys to pacman repo
- Same pattern as TTheme: versioned package built by CI, served via the terra pacman repo

## Open Questions

1. **Multi-source strategy**: The proposed "Layered Sources" approach assigns each extracted color to a different role family (primary/secondary/tertiary hue slots). Does this feel right, or should the assignment be more nuanced (e.g., picking the best color per role regardless of family)?

2. **Number of dominant colors**: 3 seems right (maps to primary/secondary/tertiary). Should we extract more and keep extras for future use? Or add a `--sources` flag to let users control it?

3. **material-color-utilities dependency**: Use the existing Python package (`pip install material-color-utilities`) or vendor the algorithm (Apache 2.0 licensed)? Vendoring avoids a runtime pip dependency but adds maintenance burden. What's the preference?

4. **Output targets**: ghostty is dropped. Any others you want to cut? pywalfox? materialized-web? tshell.toml?

5. **The "drawn" palette type**: What does this mean exactly? High-contrast with warm tones? Reduced chroma? Stylized/illustrative feel? This needs a concrete definition before implementation.

6. **Distribution**: How should the Python package be installed? Options:
   - Arch PKGBUILD with `pip install` in the package function (requires pip at install time)
   - Arch PKGBUILD with `python-build`, `python-installer` (uses pyproject.toml properly)
   - System package that depends on `python-material-color-utilities` and friends from Arch repos (if available)
